Secure Cloud File Storage System

Project Description

The Secure Cloud File Storage System was a semester long, cyber security focused, project designed and developed to cap off my Master's of Computer Engineering at the University of Louisville. I worked with tenured professor Adrian Lauf who specializes in embedded system design and cyber security. Dr. Lauf also showcased his mean cello skills during our May 2020 commencement ceremony.

Below you'll find a synopsis of the project. I highly encourage you to read my detailed engineering report containing in-depth implementation details alongside measured performance testing results.

System Requirements

Client-side development: Python 3.7 and a working computer

Server-side development: Python 3.7 and an HP Proliant MicroServer using VMWare VSphere to host a 128GB Ubuntu 19.10 Desktop distribution

Python Libraries: Sys, Time, Socket, Pathlib and Crypto*

Stage 1: File Splitting

Overview

Simply put, a user provided .txt file is split into 3 separate, but equal, .txt files.

Details

During execution, a user is prompted with a menu for the file they would like to provide. For example, a user will provide the file TestText.txt that contains This is for demo purpose only!

The client system creates a TestText.txt directory then creates and places TestText1.txt, TestText2.txt and TestText3.txt into it.

TestText1.txt receives the original text fragment of This is

TestText2.txt receives the original text fragment of for demo

TestText3.txt receives the original text fragment of purpose only!

Stage 2: File Encryption

Overview

Simply put, each file receives one of 3 unique encryption aglorithms: AES-128, RSA key pair and SHA-512/256.

Details

TestText1.txt has the AES-128 encryption algorithm applied to its contents. The user is prompted for a custom key. The key can be of 16-bit, 24-bit or 32-bit length.

TestText2.txt has the RSA key pair encryption algorithm applied to its contents. Public and private CMS (S/MIME) files and an RSA public private key pair are generated. The CMS files are loaded with their respective public private RSA keys. A random 16-bit session key is generated and encrypted via the RSA cipher. The, now encrypted, session key is used as the 16-bit key to perform AES-128 enryption on the contents of TestText2.

TestText3.txt has the SHA-512/256 encryption algorithm applied to its contents. The unencrypted contents are locally stored in a separate file. The contents are then sent through a SHA512 hash algorithm that is truncated to 256-bit length to avoid length attacks.

Stage 3: File Transfer & Storage

Overview

Simply put, the encrypted files are sent to the server one by one then combined into one file and stored.

Details
Send Files to Server

A socket is created and directed to the server's ip address on port 8888. The client sends a total of 3 messages and the server responds to each message with a status message.

The first message sends the user's selection of store.

The second message sends the size of the file's contents in bytes.

The third message sends the data of the files contents. Server-side, a buffer is created based on the file's size and a while loop is utilized to catch all packets until the buffer size is less than the size of the data received.

The files are then combined into a singular file and stored in a directory named after the file's name. If the directory does not exist, the directory is created. All 3 files, with their contents, are deleted on the client system. Necessary encryption files are kept on the client system for later decryption.

Receive Files from Server

When a user requests to get their file back from the server, the send process is repeated in reverse.

When the client receives the data for each text file, the files are re-generated on the client system with their respective contents. Upon successful send, the server deletes the combined and the file directory.

Stage 4: File Decryption & Presentation

Overview

Simply put, the encrypted files are decrypted then recombined into the original file.

Portfolio Website Pipeline

CI Tools: Git, GitHub, GitHub Actions & AWS IAM

CD Tools: AWS S3, AWS CloudFront, AWS Route53 and AWS Lambda

Description: This simple pipeline serves the static website you're currently viewing. The AWS S3 bucket is automatically updated, via GitHub Actions, when a push request is detected to the associated GitHub project. The AWS CloudFront global distribution is automatically invalidated, via AWS Lambda, when an update to the associated AWS S3 bucket is detected.

ELI5 Description: When I change the website on my computer, you will see the change very quickly without me touching anything.

Continuous Integration Details

Upon a push request to the main branch of the personal portfolio website GitHub project, a GitHub Action is triggered to update the AWS S3 bucket hosting the contents of the project.

The AWS S3 bucket update is serviced by an AWS IAM user specifically configured for pipeline requests. The GitHub Action utilizes the AWS IAM user's credentials to gain access to and update the contents of the AWS S3 bucket.

The GitHub Action sends the below command to the AWS S3 bucket. The sync command will copy new and updated files from the source GitHub project to the AWS S3 bucket. The --delete switch will delete any files that exist in the destination AWS S3 bucket and NOT in the source GitHub project.

run: aws s3 sync ./public/ s3://www.jdfordjr.com --delete

Continuous Deployment Details

Utilizing AWS S3 and AWS Route53, this website is currently hosting HTTP and HTTPS versions alongside rerouting jdfordjr.com to www.jdfordjr.com. Meaning there are 2 AWS CloudFront distributions to account for.

Using AWS Lambda, we can create a dynamic lambda function to automatically invalidate each AWS CloudFront distribution upon the associated AWS S3 bucket receiving an update. One lambda event for HTTP and one lambda event for HTTPS.

Since we want to update the entire website each time the AWS S3 bucket is updated, the AWS CloudFront invalidation will simply be: /*

The AWS Lambda function is written in Python utilizing the boto3 library to directly interact with the AWS SDK. To target both AWS CloudFront distributions, environment variable functionality from the os library is used to access each distribution ID.
